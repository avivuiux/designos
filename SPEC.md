## עקרון-מיצוב (מתוך שיטת-העבודה של אביב, 2026-06-25)

**המערכת = שותף-חשיבה בקדמת-המשפך (אפיון), הרחק מהטעם-הויזואלי.**
הבלם של אביב = אפיון-הבעיה. החוזקה לשמר = טעם ויזואלי. AI עוזר לו היום רק בוריאציות.
לכן: **להגביר את האפיון (חולשה + הכי-מבוקש בשוק), לשמר את הטעם.** פירוט: `reference/working-method.md`.

**חידוד (2026-06-25, עם התרחבות לשלב-העיצוב):** המערכת היא **מוח-התהליך-והעקרונות** - בוחרת שיטות, מיישמת חוקי-עיצוב כביקורת, מבנה תהליך. **ההכרעה האסתטית הסופית נשארת של אביב.** ("AI without replacing creatives".)

## צורת-המוצר + ארכיטקטורה (הוכרע 2026-06-26)

**צורת-המוצר = "צורה ב'": מנוע ב-IDE + סביבה-ויזואלית מעליו** (`runs/2026-06-26-canvas/06-product-form.md`).
- **המנוע** (סוכנים/מתודות/צינור-האפיון, ב-Claude Code) — מייצר תוכן מובנה. *קיים ועובד.*
- **הסביבה-הוויזואלית** (קנבס/workspace מעליו) — קוראת את התוכן, מציגה אותו כ-7 חדרי-עבודה, ומאפשרת לנוע / להריץ-מתודות / להכריע. **אופיינה** (`runs/2026-06-26-visual-environment/`), טרם נבנתה. אין אימון-מודל; רץ על המנוי (לא API נפרד).

**7 שכבות-העבודה (החדרים)** — נגזרו מהתהליך-העיצובי הקלאסי (`runs/2026-06-26-canvas/05-spec-v2.md`):
ש-1 בריף/מטרה (חוצה) · ש-2 חקר/תקיעה (קנבס-חופשי) · ש-3 הבנה/פירוק (שכבה ← פלט פירוק-משימה) · ש-4 סקיצות/כיוונים (שכבה ← פלט insight-mapper) · ש-5 מסכים/תוצר (רשימה) · ש-6 פידבק (חוצה) · **ש-7 תצוגת-התובנות (חוצה — הלב).**

**עקרונות-מוצר שננעלו (2026-06-26):**
- **ניתוב-שיטה = יכולת-עזר ברמת-הצומת (תוקן 2026-06-29, *לא* הלב):** מציעה + עוזרת-להריץ מתודה כשמזוהה פער בצומת, **כולל ריסון "מתי לא-להריץ".** מאגר-השיטות אינקרמנטלי (depth-before-breadth). הלב = ארגון-מרחב-ההחלטה + עיגון (PRODUCT §7/§11).
- **SaaS — ברור, לא מעורפל:** מוצר ממשי/מובן/אפורדנס-ברור, לא צעצוע-מחקר. חדרים+תצוגת-תובנות סרוקים, מבנה-חוזר.
- **מודל-פריט = מקום-קנוני-אחד + עדשות/קישורים** (Figma-components), אף-פעם לא עותקים.
- **השוואה-לדיפולט:** ערך שכבת-העיצוב = כמה הכיוון-המעוגן מכה עיצוב-דיפולטיבי-גנרי (עיגון-בבעיה, לא עליונות-אסתטית).
- **מדידה:** מצב-פתיחה/baseline-עיוור → **תזוזה + שׂרידוּת** (לא הצהרת-"טוב" gameable).

## חזון-המשך (north-star — חלקו כבר הוכרע/נבנה; ראה §צורת-המוצר + §מצב)

**הרחבה משלב-האפיון אל שלב-העיצוב (הכרעת אביב 2026-06-25).** עד כה Design OS חי בקדמת-המשפך (אפיון). החזון: להרחיב ל**שכבת-עיצוב-עמוק** - מערכת שיודעת לעצב *מותאם-עמוק למשתמש/לפרויקט*, גם **לבד** וגם **לפי השיטה שנבחרה**, ותוך כדי **להעמיק את ההבנה של הפרויקט עצמו**. נשארת תמיד **מוח-התהליך-והעקרונות**; ההכרעה האסתטית הסופית של אביב.

שרשרת: `אפיון (מגדיר את הפער) → בורר-שיטה (בוחר איך לגשת) → שכבת-עיצוב (מבצעת עיצוב מותאם לשיטה + להבנת-הפרויקט)`.

- **ניתוב-שיטה אדפטיבי (שער-הכניסה):** מעל ה-workflow הקבוע - מערכת שבוחרת את *התהליך הנכון לפי הבעיה*.
  - **בורר-שיטה** - סוכן שממפה מאפייני-בעיה → שיטות מומלצות.
  - **מאגר-ידע של שיטות-UX** (Journey Mapping, סיעור-מוחות, Kanban, usability testing, heuristics...) עם טריגרי "מתי". → **נדחה ללאחר-מכן** (החלטת אביב 2026-06-25).
- **שכבת-עיצוב-עמוק (הלב החדש של החזון):** **אוסף-סקילים** של עיצוב, שמערכת-מתזמרת מפעילה כדי לעצב מותאם-עמוק. כל סקיל = יכולת-עיצוב אחת ממוקדת.
  - **מבקר-גשטלט = רק סקיל אחד** מהאוסף (לא הכוכב): עוזר *להבין לפני שמעצבים* ומיישם חוקי-גשטלט כביקורת (מסמן הפרות-עקרונות, לא קובע טעם).
  - **סקילים נוספים** ייבנו ויושלבו בהדרגה (עומק-לפני-רוחב), כדי שהמערכת תדע לעצב לבד ולפי-השיטה. *(הרכב-האוסף המדויק = פתוח, ייגזר תוך כדי.)*
- **סדר:** ליבת-האפיון + הדמו מוכנים (2026-06-25) → השער נפתח. בונים אינקרמנטלית מכאן.

**שימושים-עתידיים (רעיון-אביב 2026-06-25, לכידה-לעתיד — לא v1):** כשאביב יהיה בחברה, המערכת כ**מלווה-עבודת-עיצוב יומיומית**: לאפיין דברים במוצר · לעקוב אחרי מה שעשה היום · לסכם · להציע רעיונות-שיפור + *איך* להציג אותם · להכין מהם מצגת · לקבל עדכונים. (= הרחבה מ"כלי-אפיון/עיצוב" ל"מערכת-הפעלה לעבודת-המעצב בארגון". נרשם כצפון; לא נכנס ל-scope של שכבת-העיצוב-v1 — עומק-לפני-רוחב.)

## העיקרון המכונן

**אפיון = גישור הפער בין מה שקיים לבין המטרה.**
לכן המערכת *context-grounded*: קוראת את הקיים + מבררת יעדים, לפני שהיא מציעה. בלי זה = צעצוע.

## שתי שכבות (חייבים את שתיהן)

- **שכבה 1 - מה הסוכנים עושים:** עבודת ה-UX (workflow האפיון).
- **שכבה 2 - איך נראית חווית-הסוכנים (AX):** איך המשתמש שולט, סומך, מתזמר, מאשר.

## ה-workflow הראשון: "אפיון-עומק"

brief / חומר-קיים -> אפיון מלא. **רוסטר-הצינור = שישה סוכנים (כולם נבנו ונבדקו-חי):**

1. **קולט-הקשר** - קורא קיים -> מפת-מצב. ← *הדבר-האחד שאביב בחר. הסוכן הבא לבנות. נבדק על PriceMatch.*
2. **מברר-מטרות** - יעדים / מדדים / אילוצים.
3. **חוקר-רקע** - מחקר תחום/מתחרים (web). ← *נבנה. שורשר ל-RUN כשלב 2ב (אחרי המטרות, לפני השאלות); תקציר-הרקע מקופל לשער 2 הקיים - בלי שער רביעי.*
4. **חופר-שאלות** - שאלות-העומק של מעצב-בכיר. ← *לב-הבידול. נבנה ראשון (crawl).*
5. **מאפיין** - מסנתז לאפיון מובנה (user stories, תנאי-קבלה, edge cases, מדדים).
6. **מבקר-אפיון** - יריב: "מה חסר? איזו הנחה לא אומתה?" (לולאה).

**שתי רשימות נפרדות (חשוב לא לבלבל):**

- **רוסטר-צינור-האפיון = 6** (01-06 לעיל) - מה ש-`/deep-spec` מתזמר.
- **רוסטר-Design-OS-המלא = 7** - הששה + **`07-insight-mapper` (יצרן-תובנות-עיצוב)**, שהוא **לא** חלק מצינור-האפיון אלא הסוכן-המייצר הראשון של **שכבת-העיצוב** (`north-star`): קלט=פלט-deep-spec → פלט=2-3 כיוונים-עיצוביים חלופיים. ראה `מצב` למטה.
