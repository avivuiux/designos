# הכרעה יסודית — צורת-המוצר של DesignOS

> צצה כששאל-אביב "מה המוצר שלי בעצם?" באמצע אפיון-השכבות (2026-06-26). זו הכרעה שמעל אפיון-השכבות — היא מגדירה את הצורה שהשכבות משרתות.

## ההכרעה
**DesignOS = צורה ב': מנוע ב-IDE + סביבה-ויזואלית מעליו.**
- **המנוע** (סוכנים / מתודות / צינור-האפיון) **נשאר ב-IDE** (Claude Code) — מייצר תוכן מובנה. *זה מה שכבר קיים ועובד.*
- **סביבה-ויזואלית** (הקנבס/workspace) **קוראת** את התוכן ונותנת לאביב **לראות ולנוע** בין שכבות-העבודה.
- **כרוכים זה-בזה:** המנוע **ממלא** את החדרים, הסביבה-הוויזואלית **מציגה** אותם.

## עובדות שהובהרו (לתיעוד)
- **אין אימון-מודל.** המערכת משתמשת ב-Claude קיים; האינטליגנציה במתודות/בפרומפטים.
- **API:** לא נדרש לחלק-ה-IDE (Claude Code כבר עם גישה). חלק-הוויזואלי, אם ייבנה כ-web עצמאי, יצטרך API — אך בצורה ב' הוא חזית מעל מנוע-IDE, לא אפליקציה-עצמאית.
- DesignOS היום = אוסף סוכנים/מתודות ב-IDE שמייצר מסמכים. אין GUI עדיין — הסביבה-הוויזואלית היא הבנייה הבאה.

## למה זה גורם למבנה-השכבות "לשבת"
עכשיו הכל מתחבר:
- **שכבות-העבודה (ה-7)** = החדרים שהסביבה-הוויזואלית תציג.
- **המתודות (במנוע)** = המנועים שממלאים כל חדר (פירוק-משימה→חדר-ההבנה; מפת-תובנות→חדר-התובנות).
- **הסביבה-הוויזואלית (הקנבס)** = איך רואים ונעים בין החדרים.
מיפוי: מנוע (פנימי) ממלא → תוכן חי בשכבות-עבודה → סביבה-ויזואלית מציגה.

## השלכה להמשך
ה-spec הבא הטבעי = **הסביבה-הוויזואלית עצמה** (איך רואים/נעים בין 7 החדרים, איפה הקנבס נכנס, איך מפעילים מתודה והפלט נוחת בחדר). שכבות-העבודה (`05-spec-v2.md`) = התוכנית-האדריכלית שלה.
